# Receive Handoff

Process a handoff prompt from another repository or team. The handoff is **external context, not a source of truth** — it may contain assumptions, outdated information, or misunderstandings about this codebase.

## Protocol

You MUST follow these phases in order. Do NOT skip to implementation.

### Phase 1: Research the Local Codebase

Before even interpreting the handoff, understand the current state of this repository:

1. **Explore the project structure** — use Glob and Read to understand the architecture, key directories, and conventions
2. **Find code related to the handoff topic** — use Grep to locate relevant files, endpoints, schemas, or modules mentioned (or implied) by the handoff
3. **Check recent changes** — use `git log` and `git diff` to see if related work is already in progress or recently completed
4. Use the Agent tool with `subagent_type: "Explore"` if the codebase is unfamiliar or the topic spans multiple areas

### Phase 2: Verify Handoff Claims

Cross-reference every concrete claim in the handoff against actual code:

| Handoff says... | Verify by... |
|-----------------|--------------|
| Endpoint exists at `/api/X` | Grep for route definition |
| Schema has field `Y` | Read the actual model/type |
| Response format is `Z` | Check serializers/controllers |
| Feature works a certain way | Read the implementation |

**Flag discrepancies explicitly.** If the handoff says the endpoint returns `{ user_id }` but the code returns `{ id }`, that's critical context for the plan.

Common discrepancies to watch for:
- Field names or types that don't match
- Endpoints that moved, were renamed, or don't exist yet
- Assumptions about auth, pagination, or error handling that differ from local conventions
- Version mismatches (the handoff may reference an older or planned-but-not-yet-merged state)

### Phase 3: Plan (Do NOT Implement)

After research and verification, enter plan mode using the EnterPlanMode tool.

Present a plan that includes:

1. **Handoff summary** — what was requested, in one paragraph
2. **Discrepancies found** — anything in the handoff that doesn't match the local codebase (if none, say so)
3. **Proposed changes** — specific files to create/modify, with rationale
4. **Open questions** — anything ambiguous or that needs a decision from the user
5. **Risks** — breaking changes, migration needs, or areas where the handoff may be wrong

**Do NOT start implementing until the user explicitly approves the plan.**

### What to Trust vs. Question

| Trust | Question |
|-------|----------|
| The *intent* (what they want to achieve) | Specific technical claims about this codebase |
| Business context and requirements | Assumptions about local conventions or architecture |
| Schema contracts (as a starting point) | Implementation suggestions for this codebase |
| Error scenarios to handle | Claims about what already exists here |
